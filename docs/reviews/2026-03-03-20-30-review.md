## Detailed Analysis

### Security

**Verdict: Needs Fixes**

The diff contains code relevant to security review. The primary concern is the `innerHTML` / template literal pattern in `main.ts` (`renderTable` and `renderSummary`), which interpolates site data fields without HTML escaping. While the data source is currently static JSON, the exercises explicitly encourage learners to modify site data and add their own entries -- moving this from theoretical to practical XSS risk.

**Key findings:**
- `innerHTML` with unescaped data in `renderTable` (Important) -- contrast with `exercises.ts` which correctly uses DOM APIs throughout
- localStorage `JSON.parse` without schema validation (Minor)
- Exercise IDs from `dataset.id` used as object keys (Minor)
- Missing `.env` in `.gitignore` -- relevant since exercise 10-5 teaches about API keys (Minor)

**Strengths:** `exercises.ts` uses DOM APIs correctly throughout; no credentials/secrets in code; TypeScript strict mode enabled; `loadProgress()` wraps JSON.parse in try/catch; confirm() dialog before destructive reset; exercise 10-5 teaches secrets management.

### Performance

**Verdict: Needs Fixes**

Two Important findings represent real UX issues: full DOM teardown on every checkbox tick (43 cards rebuilt, destroying focus and scroll position), and repeated synchronous localStorage reads per interaction. The missing debounce on search input is straightforward to fix.

**Key findings:**
- Full `replaceChildren` re-render on every checkbox toggle (Important)
- Multiple localStorage reads per user interaction (Important)
- No debounce on search input event (Important)
- Exercise module loaded eagerly even if panel never opened (Minor)

**Strengths:** Event delegation used correctly; `replaceChildren` over `innerHTML` for exercise list; CSS transitions for animations (off-main-thread); strict TypeScript; no external UI library dependencies.

### Correctness & Logic

**Verdict: Needs Fixes**

The intentional off-by-one bug is correctly placed for the teaching exercise. Two real issues require attention: the unhandled clipboard API rejection (silently breaks copy button in HTTP or permission-denied contexts) and the NaN persistence loop in localStorage level tracking.

**Key findings:**
- Off-by-one in `calculateStats` (Intentional -- preserved for Level 5)
- Clipboard API `.catch()` missing (Important)
- Collapsed-state persistence bug -- user cannot persist "all collapsed" (Important)
- NaN persistence loop in `loadExpandedLevel` (Minor but can permanently break accordion)
- Dead `!target.closest("input")` guard (Minor)

**Strengths:** Correct null handling for CSAT; defensive localStorage wrapping; event delegation; `replaceChildren` over innerHTML; `TOTAL_EXERCISES` computed, not hard-coded; duplicate reset button guard; strict TypeScript.

### Testing

**Verdict: Insufficient Coverage**

Zero test files exist. No test framework configured. ~550 lines of logic-bearing TypeScript with no coverage. The intentional off-by-one bug has nothing protecting it from accidental correction. localStorage handling has multiple silent failure paths untested.

**Priority test targets:**
1. `calculateStats` -- protect the intentional bug with an explicit assertion
2. `loadProgress` / `loadExpandedLevel` -- error paths and edge cases
3. `getCsatClass` -- boundary values
4. `getCompletedCount` / `findFirstIncompleteLevel` -- state combinations
5. `createExerciseCard` -- proTip presence/absence, difficulty labels
6. Delegated click handler -- accordion toggle and checkbox behavior
7. `TOTAL_EXERCISES` -- value assertion and uniqueness of exercise IDs

**Recommendation:** Add Vitest (natural fit with Vite) + jsdom for DOM tests.

### API Design

**Verdict: Needs Fixes**

Well-structured two-module architecture with clean data/logic separation, but several interface design issues: undeclared `calculateStats` return type, mutable exported `levels` array, hard-coded exercise count in HTML diverging from computed constant, unencapsulated module-level state.

**Key findings:**
- Mutable exported `levels` array (Critical)
- No declared return type on `calculateStats` (Critical)
- Hard-coded "43" in HTML vs TOTAL_EXERCISES (Important)
- localStorage keys not grouped/exported (Important)
- `renderExerciseList` hidden side effect (Important)
- Module-level mutable `expandedLevel` (Important)

**Strengths:** `ProgressMap` type alias; data/rendering separation; `DIFFICULTY_LABELS` record with type key; `loadProgress()` try/catch; `TOTAL_EXERCISES` computed constant; `findFirstIncompleteLevel` for smart UX default.

### Accessibility

**Verdict: Barriers Found**

Five Critical issues represent genuine barriers: missing form labels, icon-only button without accessible name, missing `aria-expanded` on panel toggle and accordion buttons, no focus management on panel open/close. Screen reader users cannot determine the state of interactive controls, keyboard users cannot reach the panel after opening it.

**Key findings:**
- Missing `aria-expanded` on panel toggle and accordion buttons (Critical x2)
- No focus management on panel open/close + no Escape key (Critical)
- Missing form labels on search/filters (Critical)
- Icon-only copy button without `aria-label` (Critical)
- Progress bar lacks ARIA roles (Important)
- Focus destroyed on full re-render (Important)
- Insufficient color contrast on subtitle and panel description (Important x2)
- Color-only information encoding for CSAT and status (Important x2)

**Strengths:** Semantic button elements used correctly throughout (no clickable divs); correct `<details>`/`<summary>` for hint disclosure; `<label>` wrapping checkbox + title; `lang="en"` declared; semantic landmark structure; `button.type="button"` explicit; no positive `tabindex` values.
